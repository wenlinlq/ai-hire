# 本次修改代码详解

## 1. 这次改动总体在做什么

这次修改的核心目标，是把原来更偏“静态页面展示”的前端，升级成一个具备**真实用户注册、登录、登录态保存、受保护页面访问、后端用户接口、MongoDB 持久化**的基础账号系统。

如果用一句更容易理解的话来概括：

> 以前页面大多只是长得像“招聘平台”，现在开始真正具备“用户能注册、登录、被识别、访问受限页面”的能力了。

这次改动同时涉及：

- 前端页面和路由
- 前端 API 请求封装
- 后端 Express 接口
- MongoDB 数据库连接和用户模型
- JWT 登录认证
- 环境配置和依赖安装
- 一份数据库设计文档

## 3. 这次修改的主线流程

为了先建立整体感觉，建议先看这条链路：

1. 用户在前端登录页输入用户名和密码。
2. 前端调用 `userApi.login()`。
3. `userApi.login()` 再通过 `axios` 向后端发送请求。
4. 后端 `POST /api/users/login` 收到请求后，交给控制器处理。
5. 控制器去数据库里查用户，并校验密码。
6. 如果正确，就生成 JWT token，并把用户信息和 token 返回给前端。
7. 前端把 token 和用户信息存进 `localStorage`。
8. 之后访问需要登录的页面时，前端会先检查本地是否有 token。
9. 如果没登录，就跳回登录页；如果已登录，就允许访问。
10. 以后前端再请求需要认证的接口时，会自动把 token 放到请求头里。

这就是这次改动最核心的“登录认证闭环”。

## 4. 前端部分详细说明

## 4.1 `client/package.json`

这里新增了一个依赖：

- `axios`

它的作用是专门用来发 HTTP 请求。  
以前前端页面主要还是界面逻辑，现在加入 `axios` 以后，前端就能更规范地和后端接口通信。

对应地：

- `client/pnpm-lock.yaml` 也更新了

这只是因为安装了新依赖，不是额外写了新的业务逻辑。

---

## 4.2 `client/src/api/api.ts`

这个文件是**前端请求层的基础封装**。

它做了几件事：

### 1. 创建了一个 axios 实例

```ts
const api = axios.create({
  baseURL: "http://localhost:3000/api",
  timeout: 10000,
  headers: {
    "Content-Type": "application/json",
  },
});
```

意思是：

- 所有请求默认都发到 `http://localhost:3000/api`
- 超时时间是 10 秒
- 默认使用 JSON 格式

这样做的好处是，后面每个接口都不用重复写一堆相同配置。

### 2. 添加了请求拦截器

请求发出去之前，会先检查浏览器本地有没有 token：

- 如果有，就自动放进 `Authorization` 请求头
- 格式是 `Bearer token值`

这一步很重要，因为后端很多接口会依赖 token 判断“你是不是已经登录的用户”。

### 3. 添加了响应拦截器

收到响应后：

- 成功时，直接返回 `response.data`
- 失败时，统一打印错误日志并把错误抛出去

这样上层页面就可以更简单地拿到后端真正返回的数据。

### 初学者可以这样理解

`api.ts` 不是某个具体业务接口，而是一个“统一的网络请求工具箱底座”。

## 4.3 `client/src/api/userApi.ts`

这个文件是在 `api.ts` 的基础上，专门封装“用户相关接口”。

它是前端和后端用户系统之间的桥梁。

### 里面定义了几类 TypeScript 类型

- `LoginParams`
- `RegisterParams`
- `User`
- `LoginResponse`
- `RegisterResponse`

这些类型的意义是：

- 约束传参结构
- 约束返回结果结构
- 让编辑器提示更清晰
- 减少字段写错的概率

### 它封装了哪些功能

#### 1. `login`

作用：

- 调用后端登录接口
- 登录成功后，把 token 存入 `localStorage`
- 再把用户信息也存入 `localStorage`

这意味着前端之后就能知道：

- 当前是否已登录
- 当前登录的是谁

#### 2. `register`

作用：

- 调用后端注册接口
- 注册成功后返回用户信息和提示消息

#### 3. `logout`

作用：

- 删除本地保存的 token
- 删除本地保存的用户信息

也就是“前端层面的退出登录”。

#### 4. `getCurrentUser`

作用：

- 从 `localStorage` 取出当前登录用户信息

这能让页面刷新后，前端依然知道“是谁登录了”。

#### 5. `isLoggedIn`

作用：

- 通过本地是否存在 token，判断用户是否已登录

#### 6. 其他用户管理接口

这个文件还封装了：

- `getUsers`
- `getUserById`
- `updateUser`
- `deleteUser`

这些接口更偏“用户管理后台”或“后续扩展能力”，说明这次不只是做登录注册，还顺手把用户 CRUD 的前端调用方式也准备好了。

### 这层代码的意义

如果以后任何页面要操作用户，不需要自己手写 `axios.post(...)`，直接调用 `userApi.xxx()` 就行。  
这会让代码更集中，也更容易维护。

## 4.4 `client/src/components/site.tsx`

这个文件原本主要是网站顶部导航和底部页脚。  
这次它最大的变化是：**导航栏开始根据登录状态动态变化**。

### 具体改了什么

#### 1. 新增了登录状态判断

组件里新增了：

- `useState`
- `useNavigate`
- `userApi`

用来做三件事：

- 判断当前是否登录
- 读取当前用户信息
- 退出登录后跳转页面

#### 2. 顶部菜单不再固定显示“个人中心”

原来的 `navItems` 里有：

- 首页
- 招新大厅
- AI模拟面试
- 个人中心

现在 `个人中心` 被从固定导航项里移除了。

原因是：

- 没登录时，不应该把用户引导到个人中心
- 登录后更合理的交互是放在头像下拉菜单里

#### 3. 未登录时，右侧还是“登录 / 注册”按钮

这和传统网站一致：

- 没登录，显示两个入口

#### 4. 已登录时，右侧变成用户头像和下拉菜单

已登录时，导航栏会显示：

- 一个圆形头像
- 用户名
- 下拉菜单

下拉菜单里有：

- 个人中心
- 退出登录

这里的头像不是上传图片，而是用用户名首字母生成的简易头像。

### 退出登录的处理

点击退出时会执行：

1. 调用 `userApi.logout()`
2. 关闭下拉菜单
3. 跳转回首页

### 这部分的意义

这一改动说明前端已经不再是“静态导航栏”，而是开始具备真正的“用户状态感知能力”。

## 4.5 `client/src/components/ProtectedRoute.tsx`

这是这次改动里非常关键的新组件。  
它负责“保护页面”，也就是俗称的**路由守卫**。

### 它解决什么问题

有些页面不应该让没登录的人直接打开，比如：

- 招新大厅
- AI 模拟面试
- 个人中心
- 管理页

如果没有这个组件，用户只要手动输入 URL，就可能直接进去。

### 这个组件怎么工作的

#### 1. 先获取当前路由位置

使用了：

- `useLocation`

这样它知道用户现在想访问哪个地址。

#### 2. 检查是否登录

通过：

- `userApi.isLoggedIn()`

来判断本地有没有 token。

#### 3. 用 `isChecked` 避免检查过程中的闪动

它加了一个状态：

- `isChecked`

目的很简单：

- 先完成登录状态检查
- 再决定是否渲染页面

这是一种常见的小优化，避免组件在未确认状态前先显示错误内容。

#### 4. 未登录时做两件事

- 弹出提示，告诉用户需要先登录
- 使用 `<Navigate />` 跳转到 `/login`

#### 5. 已登录时，正常渲染子组件

也就是：

- 你有 token，就允许进入被保护的页面

### 初学者可以这样理解

`ProtectedRoute` 就像门卫。  
页面本身不负责检查身份，而是先让门卫看一眼：

- 有证件，放行
- 没证件，回登录页

这个文件负责整个前端路由表。  
这次改动的重点是把多个页面包上了 `ProtectedRoute`。

#### 需要登录的页面

- 招新大厅

这说明项目从“普通多页面网站”，开始向“带权限控制的应用”发展。

## 4.7 `client/src/page/login/index.tsx`

这是前端变化最多、也是最重要的页面之一。  
它从一个偏展示型的登录/注册页，改成了一个**真正接后端的认证页面**。

### 旧版本大概是什么状态

原本这个页面更多是：

- 有登录和注册两个 tab
- 主要管理输入框状态
- 注册流程偏向手机号验证码
- 没有真正接到后端账号系统

### 新版本最关键的变化

### 1. 引入了页面跳转和用户接口

新增：

- `useNavigate`
- `userApi`

这表示页面已经不是只负责显示表单，而是要真正发请求、处理结果，并在成功后跳转。

### 2. 注册表单结构变了

原来的注册表单字段偏向：

- 手机号
- 验证码
- 密码
- 确认密码

现在改成了更标准的账号系统字段：

- 用户名
- 邮箱
- 密码
- 确认密码
- 是否同意协议

这说明当前注册逻辑已经从“短信验证码注册思路”转成了“用户名 + 邮箱 + 密码”的普通账号体系。

### 3. 去掉了验证码倒计时逻辑

以前有：

- `countdown`
- `useEffect` 倒计时
- 获取验证码按钮

这些都被移除了。  
这代表当前版本不再做短信验证码注册。

### 4. 新增加载、错误、成功状态

页面新增了：

- `isLoading`
- `error`
- `success`

这三类状态很常见，作用分别是：

- `isLoading`：防止重复提交，同时改变按钮文案
- `error`：显示失败原因
- `success`：显示成功提示

这让页面从“只提交表单”变成了“具备完整交互反馈”的表单。

### 5. 登录逻辑变成异步请求

登录提交时，现在会：

1. 阻止表单默认刷新
2. 清空旧提示
3. 进入加载状态
4. 调用 `userApi.login()`
5. 登录成功后显示“登录成功”
6. 1 秒后跳回首页
7. 失败则显示错误信息
8. 最后关闭加载状态

### 6. 注册逻辑也变成异步请求

注册提交时，现在会先做前端校验：

- 两次密码必须一致
- 必须勾选协议

通过后再调用 `userApi.register()`。

成功后会：

- 提示注册成功
- 自动切回登录 tab
- 清空注册表单

### 7. 按钮有了禁用态

登录和注册按钮在请求中都会：

- 禁止再次点击
- 显示“登录中...”或“注册中...”

这是一种比较标准的表单交互优化。

### 这一页整体说明了什么

这页已经从“静态 UI”升级为“真正能和后端认证接口配合工作的页面”。

## 5. 后端部分详细说明

## 5.1 `server/package.json`

后端新增了几项非常关键的依赖：

- `bcrypt`
- `cors`
- `dotenv`
- `jsonwebtoken`
- `mongodb`

它们分别解决不同问题：

### `bcrypt`

用来加密密码。  
用户密码不会明文存数据库，而是先哈希处理后再保存。

### `cors`

解决浏览器跨域请求问题。  
因为前端和后端通常不是同一个端口，所以浏览器会限制跨域，必须由后端明确允许。

### `dotenv`

用来读取 `.env` 文件中的环境变量。

### `jsonwebtoken`

用来生成和验证 JWT token。  
这是本次登录认证的核心之一。

### `mongodb`

让 Node.js 可以直接连接 MongoDB 数据库。

这也说明后端存储方案从这次开始明确是 MongoDB。

对应地：

- `server/package-lock.json`
- `server/node_modules/**`

也随之更新。

---

## 5.2 `server/app.js`

这个文件是 Express 应用主入口之一。  
它负责“应用级别”的中间件和路由挂载。

### 主要变化有三点

### 1. 新增了 CORS 中间件

现在后端会执行：

- 允许跨域请求
- 允许常见的请求方法
- 允许 `Authorization` 请求头

其中允许 `Authorization` 很关键，因为前端会把 JWT token 放在这里。

### 2. 用户路由从旧地址切换到新地址

原先使用的是：

- `./routes/users`
- 挂载在 `/users`

现在改成：

- `./routes/api/users`
- 挂载在 `/api/users`

这说明接口开始走更规范的 REST 风格前缀。

比如：

- 旧思路：`/users`
- 新思路：`/api/users`

这样前后端分工会更清晰。

### 3. 旧的 `routes/users.js` 被淘汰

原来的 `server/routes/users.js` 只是一个简单示例路由：

- 返回一段固定文本

它已经不适合真实项目需求，所以这次直接删除了。

### 这一层的意义

`app.js` 现在正式从“Express 初始模板”转向“真实 API 服务”。

## 5.3 `server/bin/www`

这个文件原本是 Express 默认生成的启动脚本。  
这次它被改成了“**先连数据库，再启动服务**”。

### 以前的启动方式

大致是：

- 直接 `server.listen(port)`

这在没有数据库依赖时没问题。

### 现在的启动方式

增加了 `startServer()`：

1. 先执行 `connectDB()`
2. 再加载用户模型
3. 执行 `userModel.initIndexes()`
4. 最后才开始监听端口

### 为什么这样改

因为现在后端依赖 MongoDB：

- 如果数据库都没连上，接口即使启动成功也没法正常工作
- 所以更合理的方式是“数据库准备好以后再对外提供服务”

### `initIndexes()` 的作用

用户表会建立索引，比如：

- 用户名唯一索引
- 邮箱唯一索引
- 角色普通索引

这样可以：

- 防止重复用户名
- 防止重复邮箱
- 提高按角色查询效率

### 初学者可以这样理解

以前启动流程只有“把门打开”。  
现在变成“先把仓库和设备都准备好，再开门营业”。

## 5.4 `server/db/db.js`

这个文件是**数据库连接层**，负责管理 MongoDB 连接。

### 它做了什么

#### 1. 读取环境变量

通过 `dotenv` 读取：

- `MONGODB_URI`
- `DB_NAME`

如果没配置，就使用默认值：

- `mongodb://localhost:27017`
- `ai-hire`

#### 2. 提供 `connectDB`

作用：

- 创建 MongoDB 客户端
- 建立连接
- 选中数据库
- 保存连接实例

#### 3. 提供 `getDB`

作用：

- 让其他模块拿到当前数据库实例

#### 4. 提供 `getClient`

作用：

- 拿到更底层的 MongoDB 客户端
- 以后如果要做事务等高级操作会有用

#### 5. 提供 `closeDB`

作用：

- 关闭数据库连接

### 这一层为什么重要

它把“连接数据库”这件事单独抽出来了。  
这样模型层不需要自己反复写连接逻辑，只负责操作数据。

## 5.5 `server/middlewares/authMiddleware.js`

这个文件负责 JWT 认证相关逻辑。

### 它主要提供两个函数

### 1. `generateToken(user)`

登录成功后，会根据用户信息生成 token。

它把这些信息放进 payload：

- 用户 id
- 用户名
- 角色

然后生成一个有效期为 24 小时的 JWT。

### 2. `verifyToken(req, res, next)`

这个函数是一个 Express 中间件。

它的工作流程是：

1. 从请求头里读取 `Authorization`
2. 取出 Bearer token
3. 校验 token 是否合法
4. 如果合法，把解码后的用户信息挂到 `req.user`
5. 然后调用 `next()` 放行
6. 如果不合法，直接返回 401

### 为什么需要中间件

因为很多接口都会重复做“你有没有登录”的检查。  
把这段逻辑抽成中间件后，就不用每个接口手写一遍。

### 初学者可以这样理解

JWT 像一张临时通行证。  
`generateToken` 负责发证，`verifyToken` 负责验票。

## 5.6 `server/models/userModel.js`

这个文件是用户数据模型层。  
它负责**直接和 MongoDB 的 `users` 集合打交道**。

可以把它理解为：

- 控制器负责“业务规则”
- 模型负责“真正读写数据库”

### 主要功能

### 1. `getCollection`

延迟获取 `users` 集合。  
只有真正要查数据库时，才去拿集合对象。

### 2. `initIndexes`

初始化索引：

- `username` 唯一索引
- `email` 唯一索引
- `role` 普通索引

这是数据库层防止重复数据的重要手段。

### 3. `createUser`

它在创建用户时做了很重要的一步：

- 用 `bcrypt.hash()` 对密码加密

然后再把用户信息写入数据库。

保存的字段包括：

- `username`
- `password`（加密后的）
- `email`
- `phone`
- `role`
- `avatar`
- `status`
- `lastLogin`
- `createdAt`
- `updatedAt`

### 4. 查询相关方法

包括：

- `findUserById`
- `findUserByUsername`
- `findUserByEmail`
- `findUsersByRole`
- `findAllUsers`
- `findUsersWithPagination`

这些方法对应不同维度的查询需求。

### 5. `verifyPassword`

使用 `bcrypt.compare()` 比较：

- 用户输入的明文密码
- 数据库里保存的哈希密码

这是登录校验的关键。

### 6. `updateUser`

更新用户时会：

- 自动更新 `updatedAt`
- 如果修改了密码，会重新加密后再保存

这说明代码已经考虑到了“密码不能明文更新”的问题。

### 7. `updateLastLogin`

登录成功后更新：

- `lastLogin`
- `updatedAt`

这能帮助后续做用户活跃记录。

### 8. `deleteUser`

根据 `_id` 删除用户。

这是物理删除，不是软删除。

### 这一层的意义

模型层把所有“用户数据如何存、如何查、如何改”都封装起来了。  
这样控制器只需要说“我要创建用户”“我要查用户”，不用关心底层 MongoDB 细节。

---

## 5.7 `server/controllers/userController.js`

这是后端业务逻辑最集中的文件之一。  
它负责处理用户相关的 HTTP 请求，并调用模型层完成真正的数据操作。

### 这个文件实现了哪些接口逻辑

### 1. `createUser`

作用：

- 管理员创建用户

逻辑包括：

- 读取请求体字段
- 检查必填项
- 检查用户名是否重复
- 检查邮箱是否重复
- 创建用户
- 返回去掉密码后的用户数据

### 2. `getUserById`

作用：

- 根据用户 id 查询单个用户

如果查不到，返回 404。

### 3. `updateUser`

作用：

- 更新指定用户

逻辑包括：

- 先检查用户是否存在
- 如果改用户名，要先检查新用户名是否被占用
- 如果改邮箱，也要检查邮箱是否重复
- 再调用模型更新
- 最后返回更新后的用户信息

### 4. `deleteUser`

作用：

- 删除指定用户

也是先检查存在性，再执行删除。

### 5. `getAllUsers`

作用：

- 查询全部用户

返回前会把密码字段去掉。

### 6. `getUsersWithPagination`

作用：

- 分页查询用户列表

支持：

- `page`
- `limit`
- `role`

也就是说，这个接口已经考虑到了：

- 大量数据不能一次性全查
- 可以按角色过滤

### 7. `getUsersByRole`

作用：

- 查询某个角色下的全部用户

### 8. `login`

这是最关键的认证接口之一。

处理流程是：

1. 读取用户名和密码
2. 检查是否为空
3. 根据用户名查数据库
4. 检查用户是否存在
5. 检查账号状态是否可用
6. 校验密码是否正确
7. 更新最后登录时间
8. 生成 JWT token
9. 返回用户信息和 token

注意这里返回前也会去掉密码字段，这是很必要的安全处理。

### 9. `register`

这是注册接口。

处理流程是：

1. 读取用户名、密码、邮箱等字段
2. 检查必填项
3. 检查用户名是否重复
4. 检查邮箱是否重复
5. 创建新用户
6. 默认角色设为 `student`
7. 默认状态设为 `active`
8. 返回注册成功结果

### 这个控制器整体说明了什么

它已经不再是简单的“接收请求后直接存数据库”，而是加入了比较完整的业务判断：

- 参数校验
- 唯一性校验
- 状态校验
- 安全处理
- 统一响应结构

这说明后端已经开始具备真正应用服务的形态。

## 5.8 `server/routes/api/users.js`

这个文件是用户相关接口的路由定义层。  
它负责把“URL + 请求方法”映射到具体控制器方法。

### 公开接口

这些接口不要求先带 token：

- `POST /api/users/register`
- `POST /api/users/login`

为什么不需要 token？

因为用户本来就是来注册和登录的，这时还没有 token。

### 受保护接口

这些接口前面都加了 `verifyToken`：

- `POST /api/users/`
- `GET /api/users/`
- `GET /api/users/pagination`
- `GET /api/users/role/:role`
- `GET /api/users/:id`
- `PUT /api/users/:id`
- `DELETE /api/users/:id`

这意味着：

- 没有合法 token，就不能访问这些接口

### 为什么路由层单独存在

因为一个完整的后端一般会分三层：

1. 路由层：决定请求去哪里
2. 控制器层：决定业务怎么做
3. 模型层：决定数据库怎么操作

这种分层比把所有代码写在一个文件里更清晰。

## 7. 这次前后端是怎么配合起来的

这一部分非常适合初学者建立“全链路”理解。

### 注册流程

1. 用户在前端注册页填写用户名、邮箱、密码。
2. 前端先检查两次密码是否一致。
3. 前端再检查用户是否勾选协议。
4. 前端调用 `userApi.register()`。
5. `userApi.register()` 通过 `api.ts` 发请求到 `POST /api/users/register`。
6. 后端控制器校验参数是否完整。
7. 后端检查用户名和邮箱是否已存在。
8. 如果都通过，模型层创建用户。
9. 模型层会先用 `bcrypt` 加密密码，再写入 MongoDB。
10. 后端返回注册成功消息。
11. 前端显示“注册成功”，并切回登录页。

### 登录流程

1. 用户输入用户名和密码。
2. 前端调用 `userApi.login()`。
3. 请求发到 `POST /api/users/login`。
4. 后端按用户名查用户。
5. 后端检查账号状态。
6. 后端用 `bcrypt.compare()` 验证密码。
7. 验证成功后生成 JWT token。
8. 后端返回用户信息和 token。
9. 前端把 token 存到 `localStorage`。
10. 前端把用户信息也存到 `localStorage`。
11. 用户之后访问受保护页面时，就会被识别为已登录。

### 访问受保护页面流程

1. 用户打开如 `/profile`、`/hall` 这样的地址。
2. 路由先经过 `ProtectedRoute`。
3. `ProtectedRoute` 调用 `userApi.isLoggedIn()`。
4. 如果本地有 token，允许访问。
5. 如果没有 token，弹出提示并跳转到 `/login`。

### 请求受保护接口流程

1. 前端调用 `userApi` 中某个需要认证的接口。
2. `api.ts` 的请求拦截器自动从 `localStorage` 取 token。
3. token 被放进 `Authorization` 请求头。
4. 后端路由里的 `verifyToken` 中间件验证 token。
5. 验证通过才会进入控制器。
